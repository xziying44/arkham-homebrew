# 玩家卡自动编号排序修复设计

**问题**

内容包自动编号当前只支持按遭遇组分组、按卡牌类型排序。对于无遭遇组的玩家卡，现有逻辑缺少玩家卡专用排序规则，导致以下问题：

- 调查员不会和对应签名卡排在一起。
- 玩家卡职阶没有按固定顺序排序。
- 同一职阶内没有按等级排序。
- Bonded 卡不会跟随对应主卡。

**目标**

修复玩家卡自动编号，使其更接近官方新版扩展的排序方式，至少满足以下规则：

- 职阶固定顺序：守护者、探求者、潜修者、流浪者、生存者。
- 调查员与其签名卡保持相邻。
- 主卡与其 Bonded 卡保持相邻。
- 同职阶普通玩家卡按等级升序排序。

`中立`、`多职阶`、`弱点`、未知职阶排在五大职阶之后，并保持稳定排序。

**现状分析**

现有实现位于 `bin/card_numbering.py`：

- `group_cards_by_encounter` 只负责分组。
- `sort_cards_in_group` 只按 `type` 排序。
- `generate_numbering_plan` 对所有组统一使用同一排序逻辑。

项目中已经存在可复用的关联信息来源：

- 调查员签名卡：卡牌文件中的 `tts_config.signatures = [{ path, count }]`。
- 稳定卡牌标识：`tts_config.script_id`、`id`、文件路径。
- 玩家卡等级：`level`。
- 职阶：`class`。
- Bonded 关联：优先读取 `bonded_to` / `bonded_count`；若缺失，则允许通过主卡与从卡的显式引用字段回退。

**设计**

在自动编号阶段，为“无遭遇组卡牌”增加玩家卡专用排序层：

1. 先识别玩家卡与非玩家卡。
2. 玩家卡按职阶顺序排序。
3. 每个职阶内优先输出调查员卡。
4. 每个调查员后紧跟其签名卡。
5. 每张主卡后紧跟其 Bonded 卡。
6. 其他玩家卡按等级升序、名称、文件名稳定排序。
7. 非玩家卡继续沿用现有类型排序。

**实现边界**

- 只修改自动编号链路，不修改前端配置项和交互。
- 不新增用户可配置排序规则。
- 不改动 ArkhamDB 导入导出逻辑。

**测试策略**

新增针对 `bin/card_numbering.py` 的单元测试，覆盖：

- 固定职阶顺序。
- 调查员与签名卡相邻。
- 同职阶按等级排序。
- Bonded 卡紧跟主卡。
- 非玩家卡与现有遭遇组逻辑不回归。
